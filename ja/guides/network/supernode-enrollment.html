スーパーノードプログラムに登録する¶

このガイドでは、ノードを作成して スーパーノード として登録する方法を示します。このガイドを続行する前に スーパーノードプログラム について理解してください。

Symbol ノードの立ち上げ ガイドには Symbol ノードの作成方法の詳細が含まれており、そのコンテンツはこのガイドと 部分的に重複しています 。そのガイドに従って 通常ノード を構築するか、このガイドに従って スーパーノード を構築できますが 両方 を構築することはできません。

次の2つのセクションでは、最も簡単な方法である Symbol Bootstrap を使用 してスーパーノードを作成する方法と、 手動 で作成する方法を説明します。

注釈

スーパーノードプログラムは、現在テストネットでのみテストが行われています。まもなくメインネットでもプログラムが開始される予定です。

Symbol Bootstrap の使用¶

Symbol Bootstrap は Symbol ノードを作成して、起動するのに 最も簡単な方法 です。使用したことがない場合は このツールについての学習 してください。そのガイドに記載があるように、ノードが機能するためには、ポート 3000 、 7881 、および 7900 が開いていなければならないことに注意してください。

  1. 以下を実行して、最新の Symbol Bootstrap を使用していることを確認してください。

    npm install -g symbol-bootstrap

    バージョン 1.0.5 以上である必要があります。

  2. 例として supernode.yml と命名した カスタムプリセットファイル を作成します:

    nodes:
  - rewardProgram: SuperNode
    host: my-symbol-node.com # Could also be an IP address

    このプリセットファイルはさらにカスタマイズできます:

    • ノードを 投票ノード (かつ 投票ノード報酬プログラム を受けたい) にもする場合は、 host 行の下に voting: true を同じインデントで追加してください。

    • 監視エージェントが使用するデフォルトのポート (7881) を変更する場合は、 (例として) ファイルの先頭に rewardProgramAgentPort: 7882 を追加できます。

    • 必要なノードアカウント (main, transport, remote, VRF または voting) のいくつかをすでに持っている場合は プリセットファイルにそれらを記述することもできます 。

      注釈

      マルチシグアカウント でノードを管理する場合、上記プリセットファイルの メイン アカウントは (連署者では なく) マルチシグアカウント でなければなりません。

    Symbol Bootstrap ドキュメント でカスタマイズ可能なリストを参照してください。

  3. 新しいプリセットファイルを使用して ノードを作成して実行します :

    標準のノードサービスに加えて 監視エージェント もダウンロードしてインストールします。

    symbol-bootstrap start -p mainnet -a dual -c supernode.yml

    start コマンドはノード設定を作成し、起動を行います。

    注釈

    テスト目的に -p testnet を使うことで Symbol のテストネットを使うことができます。

  4. あなたのノードが 十分な資産 を保有していることを確認してください。 スーパーノードとして適格になる には、メインアカウントに 100 万 以上の symbol.xym を保有する必要があります。

    ノードのメインアカウントアドレスは target/addresses.yml ファイルにあります。

    テストネットでは Symbol Faucet を使用して、あなたのノードのメインアカウントに トークンを送る ことができます。もしフォーセットが提供できる以上のトークンが必要な場合は NEM’s Telegram Help Desk で cryptobeliever に連絡をください。

  5. リモート、VRF、投票キーをリンクして ノードを登録します 。

    Symbol Bootstrap を中断せずに、別のターミナルを開き、先ほどと同じフォルダに移動します。そして下記を実行します:

    symbol-bootstrap link --useKnownRestGateways

    これにより、指定した鍵をメインアカウントにリンクするトランザクションが送信されます。このトランザクションは少額の 手数料 がかかるので、メインアカウントにスーパーノードプログラムの要件を超える 余剰資金 があることを確認してください。

    注釈

    テスト期間中に重いテストを行うと、トランザクション手数料が人為的に増加する可能性があります。もし symbol-bootstrap が Announcing Simple Transaction hash... というメッセージでフリーズする場合は、例えば --maxFee 10000000 パラメータ (この絶対単位は 10 symbol.xym に対応) を追加して、トランザクション手数料を増やしてみてください。

  6. スーパーノードプログラムに登録する

    symbol-bootstrap enrollRewardProgram --useKnownRestGateways

    これにより、加入を要求するトランザクションがコントローラアドレスに送信 (前のステップの手数料に関する注釈を参照) されます。

    この時点から、ネットワーク上の 報酬プログラムコントローラ がノードの監視を始めます。

このガイドの最終セクション 検証 にジャンプできます。

手動¶

もし Symbol Bootstrap を使用しない (あるいは使用できない) 場合でも、手動でスーパーノードプログラムに加入することができます。これらの手順は Linux 向け のみ利用可能です。

はじめに、ノードが起動している (または実行できる状態にある) ことを確認します。 手動による Symbol ノードの立ち上げ ではノードのセットアップ方法を説明しています。

After following that guide, you need to install a monitoring agent and announce a special transaction. You are going to create a few files, so please create a folder to keep your filesystem tidy (for example called agent inside Catapult client’s _build folder).

  1. ノードモニタリングエージェントのダウンロード:

    Linux バイナリをダウンロード して、実行可能にします:

    wget https://symbol-node-reward.s3-eu-west-1.amazonaws.com/packages/1.0.2-alpha.0/agent_binary/agent-linux.bin
chmod +x agent-linux.bin

  2. モニタリングエージェントのための 証明書を作成 します:

    エージェントは 報酬プログラムコントローラ へのすべての接続を 認証 するため、SSL 証明書を作成する必要があります。

    OpenSSL が最低でも バージョン1.1.1 であることを確認します。例えば sudo apt install openssl を実行してインストールできます。

    • certs フォルダを作成して移動します。

    • 次の内容で agent-ca.cnf ファイルを新規作成します。

      [ca]
default_ca = CA_default
[CA_default]
policy = policy_catapult
[policy_catapult]
commonName = supplied
[req]
prompt = no
distinguished_name = dn
[dn]
CN = Agent CA

    • 次を実行して証明書を生成します:

      # Generate agent CA key pair
openssl genpkey -out agent-ca.key.pem -outform PEM -algorithm ed25519

# Create agent CA CSR
openssl req -config agent-ca.cnf -key agent-ca.key.pem -new -out agent-ca.csr.pem

# Base64 encode agent CA CSR file for later use
base64 agent-ca.csr.pem --wrap 0 ; echo

      最後のコマンドは文字列を出力します。後で使用するのでコピーします。

    • 最後に、エージェントが監視しているノードへの接続を認証するために、Symbol ネットワーク CA 証明書をダウンロード (1 つだけダウンロード) します。

      # For MAINNET
wget https://symbol-node-reward.s3-eu-west-1.amazonaws.com/mainnet/certs/controller-ca-crt.pem

# For TESTNET
wget https://symbol-node-reward.s3-eu-west-1.amazonaws.com/testnet/certs/controller-ca-crt.pem

    • certs フォルダから移動します。

  3. モニタリングエージェントの設定:

    agent.properties という名前のテキストファイルを作成し、以下の内容を追加し、 NODE_PRIVATE_KEY と REST_GATEWAY_URL を適切な値に置き換えます:

    ; Use 104 for MAINNET or 152 for TESTNET
NETWORK_TYPE=104
LOGGER_FILE=agent.log
; Replace with the public host where your node is running (hostname or IP address)
REST_GATEWAY_URL=http://my-symbol-node.com:3000
REWARD_PROGRAM=SuperNode
; Replace with the main account's public key
MAIN_PUBLIC_KEY=●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●
CERTS_CONTROLLER_CA_CERT_FILE=./certs/controller-ca-crt.pem
CERTS_AGENT_CA_KEY_FILE=./certs/agent-ca.key.pem
CONTROLLER_API_URL=http://node-monitoring.symbolblockchain.io:7890; MAINNET
; CONTROLLER_API_URL=http://node-monitoring.testnet.symboldev.network:7890; TESTNET

    注釈

    デフォルトでは、エージェントはポート番号 7881 を使って通信を行います。このポートが利用できない場合は HTTP_PORT= の行を追加して、別のポート番号を指定することができます。

    加入プロセスや後のモニタリングの際、コントローラによるノードの REST インターフェースへの接続方法を指定するために REST_GATEWAY_URL を指定する必要があります。

  4. エージェントの起動:

    このコマンドを起動し、ノードが動作している間は 起動を続けて ください。コントローラは定期的にエージェントに問い合わせを行い、エージェントが十分な回数の応答をできなかった場合、あなたのノードは スーパーノード報酬プログラムから除外される 可能性があることを覚えておいてください。

    ./agent-linux.bin --config agent.properties

    注釈

    エージェントが起動すると、登録が完了して、新しい通信証明書が生成されるまで、2 分毎にコントローラをポーリングします。この時点では、エージェントはコントローラからのコマンドを待機しているだけです。

  5. 加入メッセージの送信

    最後に、自分のノードをスーパーノードプログラムに加入させることを コントローラに通知します 。これはモザイクなしの特別なメッセージを含む従来の 転送トランザクション で行います。

    enroll AGENT_URL BASE64_ENCODED_AGENT_CA_CSR
    • AGENT_URL を https:// + エージェントを動作させているホスト + :7881 に置き換えてください。この URL は どこからでもアクセスできる ものでなければなりません。例えば、https://my-symbol-node.com:7881 のようになります。IP アドレスも有効です。デフォルトのポート番号を使用していない場合は、上記ステップ 3 で指定したポート番号を使用してください。
    • BASE64_ENCODED_AGENT_CA_CSR をステップ 2 の出力で置き換えます。

    最後に、このトランザクションの受信アドレスは:

    • NDG2F6IHON7EDOXZCHSTSJ2YMUHDFXAQ2EUZHFA メインネット用
    • TDL73SDUMPDK7EOF7H3O4F5WB5WHG2SX7XUSFZQ テストネット

    symbol-cli でトランザクションをアナウンスします:

    symbol-cli transaction transfer --mode normal --sync \
           --recipient-address NDG2F6IHON7EDOXZCHSTSJ2YMUHDFXAQ2EUZHFA \
           --message "enrol AGENT_URL BASE64_ENCODED_AGENT_CA_CSR" \
           --mosaics @symbol.xym::0

    このトランザクションはノードのメインアカウントによって署名されている必要があります 。したがって、これが symbol-cli のデフォルトプロファイルであることを確認してください。

    このトランザクションでは少額の 手数料 を支払うため、メインアカウントにスーパーノードプログラムの要件を超える 余剰の資産 があることを確認してください。

    注釈

    もし symbol-cli が使用するデフォルト手数料が、現在のネットワーク状況に対して小さすぎる場合、プログラムが トランザクション送信 ステップの処理中にスタックする場合があります。

    コマンドが 5 分以内に終了しない場合は、 Ctrl+C を押し、 1 XYM* の手数料を設定する --max-fee 1000000 パラメータを追加して、再試行します。 手数料 を読んでいるなら、自由に任意の数値を設定してもよいです。

セットアップの検証¶

すべてのサービスが正しく動作していることを確認して ノードを検証 できます。次の URL で有効なデータを返却することを確認できます:

  • http://localhost:3000/chain/info: ノードの接続
  • http://localhost:3000/node/info: ノードの状態
  • https://localhost:7881/metadata:エージェントレポート (下記参照)

次に、パブリックホスト名からアクセスできることをもう一度確認します。

注釈

最後のエンドポイントにアクセスするには、TLS チェックを無効にできるコマンドラインツールを使用します。例:

curl --insecure https://localhost:7881/metadata

もしくは:

wget --no-check-certificate https://localhost:7881/metadata

加入が完了したら、 Symbol Explorer (MAINNET) (または TESTNET) で、自分のノードがリストに表示され、適切な報酬プログラムのマーク (情報は 30 秒ごとに更新) が表示されていることを確認できます。
