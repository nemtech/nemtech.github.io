    交易¶

    交易通常代表数据库系统中的工作单元。在区块链的定义中，表现为经由 账户 签名来改变账户状态的操作。

    被网络接受的交易将被长久存储在 区块 中。

    交易类型¶

    交易的类型很多。例如，您可以在帐户之间转移 马赛克 ，转移或配置帐户的所有权（包括使用 多重签名 规则）等等。

    Id 类型 说明
    马赛克    
    0x414D Mosaic Definition Transaction 注册一个新的 马赛克.
    0x424D Mosaic Supply Change Transaction 改变现有马赛克的供应量。
    命名空间    
    0x414E Register Namespace Transaction 注册一个 命名空间 来组织你的资产。
    0x424E Address Alias Transaction 为账户附加命名空间。
    0x434E Mosaic Alias Transaction 为马赛克附加命名空间。
    转让    
    0x4154 Transfer Transaction 在两个账户之间发送马赛克或信息。
    多重签名    
    0x4155 Modify Multisig Account Transaction 创建或修改 多重签名合约.
    0x4141 Aggregate Complete Transaction 将交易批量发送到不同的帐户。
    0x4241 Aggregate Bonded Transaction 汇总在不同帐户间的许多交易。
    0x4148 Hash Lock Transaction 公布保税聚合交易之前的押金。
    – Cosignature Transaction 共签保税聚合交易。
    账户过滤器    
    0x4150 Account Properties Address Transaction 允许或阻止一组特定地址所转入的交易。
    0x4250 Account Properties Mosaic Transaction 允许或阻止包含特定马赛克的转入交易。
    0x4350 Account Properties Entity Type Transaction 按交易类型允许或阻止转出交易。
    跨链交换    
    0x4152 Secret Lock Transaction 在不同的链之间开启 代币交换。
    0x4252 秘密证明交易 结束跨链的代币 交换。
    远程收获    
    0x414C 账户连系交易 将帐户权限委托给代理帐户以启动 委托收获。

    定义一个交易。¶

    每笔交易之间都有一些共同属性，它们都是经过 交易模式定义 拓展而来，添加了不同类型的特定属性。

    交易以 序列化形式 定义， 我们建议 使用NEM2-SDK 来定义交易。

    费用¶

    交易会产生相关成本。这个成本是必要的，以激励保护网络并运行基础设施的 收获者。

    交易相关费用主要取决于交易的规模。有效费用是交易规模和收获者设定的费用乘数的乘积。节点所有者可以将后一个值配置为任何正值，包括零。

    effective_fee = transaction::size * block::fee_multiplier

    交易的发起人必须在准备交易时指定 max_fee，这是帐户愿意接受的最大费用。

    如果 effective_fee 小于或等于 max_fee，则收割者可以选择在区块中收录该交易。fee_multiplier 存储在 区块的首部 中，为每笔交易被收录区块内所要花费的有效费用提供参考。

    收获节点可以决定其交易收录策略：

    • 先入为主：适用于具有高交易吞吐量要求的网络。优先收录最早的交易。
    • 最少的费用：慈善节点。收录其他节点不想收录的交易。
    • 最多的费用：常见于公共网络。收录较高费用的交易。

    在预设的情况下，费用是通过NEM网络的基础货币 cat.currency 进行支付。私链可以修改网络配置以取消费用，或者使用其他合适的 马赛克 。

    签署交易¶

    帐户必须在将交易公布到网络之前对其进行签名。签署交易 表示帐户同意按定义更改网络状态。

    例如，转让交易描述了谁是收件人以及要转让的马赛克数量。在这种情况下，签署交易意味着接受将这些马赛克从一个帐户的余额中移动到另一个帐户的余额中去。

    该帐户使用私钥 签署交易的前100个字节 而生成签名。 然后，签名将附加到交易的正文，从而产生一个已签名的交易。

    一旦 sha3-256演算法 被应用于序列化的交易,该交易的哈希值就产生了 。

    公布一个交易¶

    已签名的交易已准备好向网络公布。

    交易周期

    在 公布交易 后，REST API通常立刻会返回 OK 相应。但这还无从知晓交易的有效性。

    验证的第一阶段发生在API节点中。如果交易出现某些错误，WebSocket将通过状态频道发出通知。交易会以 未确认 的状态到达P2P网络。请不要将未确认的交易作为依据。至此，还不能确定该交易是否会被收录到区块中，因为它需要通过第二次验证。

    第二次验证将在交易被收录到区块中之前完成。如果交易有效，则收获者会把交易收录在区块中，该交易算是被 确认 了。

    继续前面的例子，交易得到处理且所述金额从签名者的帐户转移到收件人的帐户。除此之外，交易费会用从签名者的账户中扣除。

    此时交易有 零确认。 当另一个区块添加到区块链时，该交易有一个确认。 当再有一个区块被添加到区块链中，该交易便有两个确认。依此类推。

    回滚¶

    区块链被设计可在在某些情况下需要回滚到最近的区块。这对于解决区块链的分叉至关重要。

    The “rewrite limit” is the maximum number of blocks that can be rolled back. Hence, forks can only be resolved up to a certain depth too.

    NEM的重写限制为 360 区块。 一旦交易超过360次确认，就无法回滚。

    根据以往经验，除非代码有误或遭到攻击导致区块链出现严重问题，否则一般不会发生超过20个区块的分叉。
